2. Technical Implementation (React Stack)
Since you are using React, you have three main paths. For a PMS, Path B is usually the best balance of performance and ease of use.
Path A: HTML5 Canvas (via react-konva)
Pros: Extremely high performance (can handle hundreds of rooms/tables), easy image export, standard for graphics.
Cons: Accessibility is harder; text rendering can be blurry if not handled well.
Verdict: Best for complex layouts.
Path B: SVG + Framer Motion / React Draggable
Pros: DOM elements are easier to style with Tailwind CSS. Sharp text. Accessibility is native.
Cons: Performance drops if you have 1000+ elements (unlikely for a 3-star hotel).
Verdict: Recommended. It integrates tightest with your UI library (Shadcn).
Recommended Libraries:
dnd-kit: For the drag-and-drop logic (palette to canvas).
react-resizable or react-rnd: To allow admins to resize rooms/tables.
react-zoom-pan-pinch: To handle the zooming of the map canvas.

Based on the Electronic Billing Procedure, 2074 (4th Amendment) and current IRD directives, here are the mandatory requirements your software must meet to get approved by the Inland Revenue Department (IRD) of Nepal:
1. Core Data Integrity (Crucial)
Immutable Transactions: Once an invoice is generated, it cannot be edited or deleted.
Correction Method: Mistakes must be corrected only by issuing a Credit Note (for sales return) or Debit Note, never by altering the original record.
Audit Trail: You must log every critical action (User ID, Timestamp, Action Type) in a User Activity Log that cannot be deleted by the user.
2. CBMS Integration (Real-Time Sync)
Real-Time API: Your system must connect to the IRD's Central Billing Monitoring System (CBMS).
Sync Data: Every Sale, Sales Return, and Write-off must be synced instantly to the IRD server (using their JSON API format).
API Credentials: The software must allow entering the CBMS username/password provided by the IRD for each specific taxpayer.
3. Invoice Standards
Sequential Numbering: Invoice numbers must be sequential and reset every Fiscal Year (e.g., INV-080/81-0001).
Mandatory Fields:
Seller's Name, Address, PAN/VAT.
Buyer's Name, Address, PAN (if applicable).
Date (BS and AD).
Items, Quantity, Rate.
Tax Breakdown: Taxable Amount, Non-Taxable Amount, VAT (13%), Service Charge (10% - optional if removed by court but system must support logic), and Grand Total.
Three Copies: The system must support printing 3 distinct copies:
Customer Copy
Accounting Copy
Office Copy
Reprint Marking: If an invoice is printed again, it must automatically stamp/watermark "COPY OF ORIGINAL" on the top.
4. Reporting (Tally & Annexes)
Sales/Purchase Register: Must generate reports in the exact column format prescribed by IRD (often called "Annex 5" format).
Materialized View: You must have a feature to view/export the "Materialized View" (Sales/Purchase VAT Register) to Excel.
Tally Export: As you noted, a feature to export Sales/Purchase data to Tally XML is practically mandatory for acceptance by auditors.
5. Technical & Security
Local Data/Backup: Even if cloud-based, you must ensure data is backed up securely. If the internet fails, the system must hold data locally and sync to CBMS once online.
Database Lock: The database should not be easily accessible/editable manually by the user (to prevent tax evasion).
